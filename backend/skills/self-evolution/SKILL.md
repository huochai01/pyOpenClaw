# 失败恢复经验沉淀

把"失败后恢复成功"的执行过程，提炼成下次可直接复用的规则，写入对应技能文件和长期记忆，而不是只停留在当前会话里。

---

## 触发条件

同时满足以下三条才执行，缺一不可：

1. **明确的失败**：首次方案失败，且有具体报错或可观察到的失败信号（不是模糊的"好像不太对"）
2. **主动的切换**：你尝试了不同的方案、参数或工具，而不只是重试同一路径
3. **稳定的成功**：替代方案成功，且这个成功是可重复的，不像是偶然运气

> 偶发性网络抖动、一次性环境异常、没有找到更优方案的失败——这三类情况不触发本技能。

---

## 执行步骤

### 第一步：还原两条路径

在写任何东西之前，先在内部明确：

```
失败路径：用了什么 → 出现了什么错误信号 → 为什么失败
成功路径：改成了什么 → 为什么它有效 → 这个规律是否稳定
```

如果你说不清楚"为什么失败"或"为什么成功"，说明经验还不够清晰，先不要写入。

### 第二步：定位写入目标

按以下优先级判断要写入哪个技能文件：

- **本轮明确读取并依赖了某个 `skills/*/SKILL.md`** → 写入该技能文件 + `memory/MEMORY.md`
- **本轮没有依赖任何具体技能** → 只写入 `memory/MEMORY.md`，不要随意改其他技能

用 `read_file` 读取目标文件，了解现有内容，再决定在哪里插入或修改。

### 第三步：提炼经验（写入前的自检）

提炼出的经验必须通过以下三个问题：

1. **可识别**：下次遇到同类问题，Agent 能从经验描述中认出"这就是我遇到的情况"吗？
2. **可执行**：经验给出的是具体行动指令，而不是模糊的"注意一下"？
3. **可复用**：这条经验在未来同类任务中还适用，而不是只针对这次的特殊情况？

三个问题都是"是"才值得写入。

### 第四步：更新技能文件

在最相关的步骤附近追加或修改，写**可执行规则**，不写事故复盘。

**格式**：
```md
## 经验教训
- [YYYY-MM-DD] 当 `<首选方案>` 出现 `<具体错误信号>` 时，直接切换到 `<备用方案>`，不要重试原路径。
```

**判断插入位置**：
- 如果经验影响某个具体步骤的执行方式 → 在该步骤下方就近插入
- 如果经验是全局性的（影响多个步骤）→ 在文件末尾统一写入 `## 经验教训` 区块

检查是否已有同类条目：有则更新原条目，不要追加重复内容。

### 第五步：更新长期记忆

写入 `memory/MEMORY.md`，记录跨任务仍有价值的内容。

**格式**：
```md
## 经验教训

### YYYY-MM-DD：<一句话主题>
- 问题特征：<可识别的失败信号，不是某次任务的具体细节>
- 有效方案：<替代方案是什么，为什么更稳定>
- 下次优先：<以后应该怎么做，写成行动指令>
- 关联技能：<如果同步更新了某个 SKILL.md，注明文件路径>
```

同样检查是否重复，有则更新原条目。

---

## 写入内容的边界

| 写入技能文件 | 写入长期记忆 |
|------------|------------|
| 任务级、流程级的执行规则 | 跨任务仍然有价值的规律 |
| 首选方案与备用方案的优先级 | 当前环境下某类依赖不稳定的事实 |
| 特定错误信号 → 对应切换动作 | 用户希望 Agent 如何处理失败的偏好 |
| 哪些参数/接口在本任务场景更稳定 | 哪个技能已因真实案例被修正（留存记录） |

**永远不要写入**：
- 一次性的网络抖动或环境异常（不稳定规律）
- 用户输入原文、密钥、任何敏感信息
- 冗长的过程日志（"我先试了A，A失败了，然后我试了B……"）
- 对多个无关技能的同时修改

---

## 完成自检

写入后，对着以下标准过一遍：

- [ ] 技能文件中出现了更具体的执行规则（或确认无需更新技能）
- [ ] `memory/MEMORY.md` 中出现了对应的跨任务经验
- [ ] 没有引入重复条目
- [ ] 下次只看技能和记忆，就能在同类失败发生前优先规避它

---

## 工具使用

- `read_file` — 读取技能文件和 `memory/MEMORY.md`
- `write_file` — 修改文件（优先用 append=`true`参数 精确插入，避免覆盖全文）
- 修改的是正文内容时，不需要刷新技能快照；如果修改了 frontmatter（name/description），需要刷新
